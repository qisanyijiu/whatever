# EnglishClozeCoach 软件工程改进建议

> 基于对项目的全面审计，涵盖架构、并发、数据持久化、错误处理、测试、UI、代码质量七个维度。

---

## 一、架构与模块设计

### 1.1 TranslationJobStore 过于臃肿（严重）

**文件**：`Sources/EnglishClozeCoach/Stores/TranslationJobStore.swift`（1002 行）

单文件承担了：Job CRUD、导入管线（文件/文件夹/粘贴）、AI 评估+翻译管线、并发门控（`ConcurrencyGate`）、状态缓冲（`ProcessingBuffer`）、系统翻译协调、定时器同步、后台落盘、重试逻辑。

**建议拆分为 3~4 个文件**：

| 拆出模块 | 职责 | 预计行数 |
|---------|------|---------|
| `TranslationJobPipeline` | AI 评估+翻译的并发处理管线 | ~200 行 |
| `TranslationJobBuffer` | `ProcessingBuffer` + `ConcurrencyGate` | ~60 行 |
| `TranslationJobPersistence` | 落盘调度、保存/加载协调 | ~80 行 |
| `TranslationJobStore` | 仅保留 UI 交互、Job CRUD、导入入口 | ~300 行 |

### 1.2 ContentView 职责过载（严重）

**文件**：`Sources/EnglishClozeCoach/Views/ContentView.swift`（427 行）

`ContentView` 14 个 `@State` 变量，承担了：庆祝动画、错题记录、AI 解释获取、系统翻译编排、自动导入翻译任务。这些不是 View 层应有的职责。

**建议**：抽取 `PracticeCoordinator` ViewModel，将业务编排从 View 移出。

### 1.3 ImportView 过大（中等）

**文件**：`Sources/EnglishClozeCoach/Views/ImportView.swift`（759 行）

单个 View 同时处理：粘贴文本、本地文件、文件夹、TED 下载、字幕 URL、AI 翻译、系统翻译、安全范围书签、文件面板。建议按导入源类型拆分为多个子 View。

### 1.4 重复的 `applicationSupportDirectory()` 实现（中等）

5 个 Library 文件各自实现了一套 `applicationSupportDirectory()` 和 `defaultApplicationSupportDirectory()`：`PracticeLibrary`、`StudyDataLibrary`、`TranslationJobLibrary`、`AIProviderLibrary`、`UserAccountLibrary`。

**建议**：抽取 `ApplicationSupport` 工具类型或共享基类。

### 1.5 *Library 命名不当（低）

`PracticeLibrary`、`StudyDataLibrary` 等并非传统意义的 Library，而是持久化 Repository。建议重命名为 `*Repository` 或 `*Persistence`。

## 二、并发安全

### 2.1 ConcurrencyGate 续体泄露导致死锁（严重）

**文件**：`Sources/EnglishClozeCoach/Stores/TranslationJobStore.swift`（`ConcurrencyGate.enter()`）

```swift
func enter() async {
    if running < max { running += 1; return }
    await withCheckedContinuation { waiters.append($0) }  // BUG
}
```

如果 `Task` 在 `withCheckedContinuation` 中被取消，`CheckedContinuation` 被静默丢弃但仍在 `waiters` 数组中。`leave()` 可能在已失效的续体上调用 `resume()`，导致：
- 并发槽位永久丢失（吞吐量下降）
- 极端情况下所有槽位泄露 → 死锁

**修复**：使用 `withTaskCancellationHandler` 或 `withThrowingContinuation`，并在取消时从 `waiters` 中移除。

### 2.2 `@unchecked Sendable` 被滥用（严重）

**文件**：`Sources/EnglishClozeCoach/Stores/TranslationJobStore.swift:33`

```swift
@MainActor
final class TranslationJobStore: ObservableObject, @unchecked Sendable {
```

`@unchecked Sendable` 禁用了编译器的 Sendable 检查，而 TranslationJobStore 同时涉及：
- `@Published` 属性（MainActor 隔离）
- `ProcessingBuffer` actor（后台）
- `Task.detached` 中的处理管线
- `ConcurrencyGate` actor
- `Timer`（non-Sendable）

这导致 Sendable 问题在运行时才会暴露。**建议**：重构为显式 Sendable 一致性，将后台处理逻辑移到独立的 non-MainActor 类型。

### 2.3 四个 Store 缺少 `@MainActor` 标注（严重）

**文件**：`PracticeStore.swift`、`StudyStore.swift`、`UserSessionStore.swift`、`AIProviderStore.swift`

这四个类都是 `ObservableObject` 且有 `@Published` 属性，但都没有显式标注 `@MainActor`。目前它们仅被 SwiftUI View 访问（隐式主线程），但如果将来有任何后台代码调用这些 Store 的方法，`@Published` 变更会在错误线程传播，导致 UI 崩溃或数据竞争。`TranslationJobStore` 正确标注了 `@MainActor`，其他 Store 应统一。这是一个编译器无法帮你检查的定时炸弹。

### 2.4 ImportView 中无法取消的后台任务（中等）

**文件**：`Sources/EnglishClozeCoach/Views/ImportView.swift:626,709,739`

`translateItems()`、TED 下载、脚本下载都创建了 `Task { ... }` 但未存储引用。用户关闭 sheet 后，这些任务不会被取消，继续消耗网络资源和 actor 状态。

### 2.5 Timer 引用的生命周期管理（低）

**文件**：`TranslationJobStore.swift`

`syncTimer` 通过 `scheduledTimer` 附加到主运行循环。`deinit` 中 invalidate，但这依赖于 deinit 在 Timer 之前触发。如果 Timer 的 target（self）被提前释放，Timer 可能继续触发。

### 2.6 StudyStore 后台保存的错误被丢弃（低）

**文件**：`Sources/EnglishClozeCoach/Stores/StudyStore.swift`

`scheduleBackgroundSave()` 使用 `_ = try? lib.save(snapshot)` 吞掉了所有保存错误。持续写盘失败（如磁盘满）不会被用户感知。

### 2.4 `processItemOffMain` 中的状态竞态（低）

**文件**：`TranslationJobStore.swift`

处理管线中，`pendingItemsSnapshot` 在主线程读取，而 `flushBufferToMainActor` 在后台写入 buffer。当后台管线还在运行而 Timer 同时 flush 时，同一个 item 可能被两次标记为 evaluating。当前设计通过 buffer actor 的 drain 语义避免了重复，但这依赖调用顺序，不够稳固。

## 三、数据持久化

### 3.1 每次按键写盘的原始设计（已部分修复）

**文件**：`Sources/EnglishClozeCoach/Stores/StudyStore.swift`

`recordMistake` 和 `recordCompletion` 原本每次调用都同步写 `StudyData.json`。已改为 `scheduleBackgroundSave()` 防抖保存。但防抖逻辑在多线程下可能产生多个 `Task.detached` 同时写同一文件，没有序列化保证。

**建议**：引入 actor-based `FileWriter` 或使用 `DispatchQueue(label: "com.whatever.save")` 序列化写操作。

### 3.2 多个后台保存任务可能并发写同一文件（中等）

`TranslationJobStore.saveJobsInBackground()`、`StudyStore.scheduleBackgroundSave()`、`PracticeStore`（已回退到同步）、`AIProviderStore`（同步）各写各的文件，暂无冲突。但如果未来引入多个任务同时写 `TranslationJobs.json`，会损坏文件。

### 3.3 初始化同步读盘（低）

所有 5 个 Store 的 `init()` 都在主线程同步读取 JSON 文件。启动时一次性开销，影响可接受，但可用 `Task.detached` + 回调优化启动体验。

### 3.4 密码哈希强度不足（低）

**文件**：`Sources/EnglishClozeCoach/Services/PasswordHasher.swift`

使用 `SHA256.hash(data: Data("\(salt):\(password)".utf8))`。对本地验证可接受，但缺少 bcrypt/scrypt/argon2 的密钥拉伸。如果未来引入网络同步，必须升级。

## 四、错误处理

### 4.1 大量 `try?` 吞掉错误（严重）

**文件**：多处

| 位置 | 代码 | 后果 |
|------|------|------|
| StudyStore | `_ = try? lib.save(snapshot)` | 磁盘满静默失败 |
| PracticeStore | `try? library.save(decks)` | 保存失败不可见 |
| TranslationJobStore | `try? await Task.sleep(...)` | 重试延迟被跳过 |
| PracticeLibrary | `try? fileManager.contentsOfDirectory(...)` | 目录读取失败静默 |
| FolderTextImporter | `try? Data(contentsOf: ...)` | 文件读取失败静默 |

### 4.2 错误消息硬编码中文（严重）

所有 Store 的 `saveError`、`importError`、`authError` 等字段使用硬编码中文字符串（如 `"保存题库失败"`、`"创建用户失败"`），内嵌在业务逻辑层。虽然 `AITextServiceError` 正确使用了 `LocalizedError`，但其他错误类型不统一。

### 4.3 `try!` 强制解包（中等）

| 文件 | 行号 | 代码 |
|------|------|------|
| ClozeGenerator.swift | 96 | `try! NSRegularExpression(pattern: ...)` |
| StudyStore.swift | 53, 68, 136-137, 252, 258, 269 | `Calendar.current.date(byAdding: ...)!` |

这些在正常情况下不会失败，但一旦失败就崩溃。

## 五、测试

### 5.1 单个巨型测试文件（严重）

**文件**：`Tests/EnglishClozeCoachUnitTests/main.swift`（2621 行，61 个测试）

所有测试在一个文件中，使用自定义测试运行器（非 XCTest）。问题：
- 难以导航和维护
- 无法并行执行测试
- 无法在 Xcode Test Navigator 中查看
- 测试失败时只能靠行号定位

**建议**：迁移到 `.testTarget` 使用 XCTest，按模块拆分为多个测试文件：
```
Tests/
  EnglishClozeCoachTests/
    Models/
      TranslationJobTests.swift
      PracticeItemTests.swift
      ImportDraftTests.swift
    Services/
      ClozeGeneratorTests.swift
      AnswerMatcherTests.swift
      AITextServiceTests.swift
      QuestionImporterTests.swift
      FolderTextImporterTests.swift
      PracticeLibraryTests.swift
    Stores/
      PracticeStoreTests.swift
      StudyStoreTests.swift
      TranslationJobStoreTests.swift
      AIProviderStoreTests.swift
      UserSessionStoreTests.swift
      AICompletionClientTests.swift
```

### 5.2 Package.swift 中使用 `executableTarget` 做测试（中等）

```swift
.executableTarget(
    name: "EnglishClozeCoachUnitTests",
    ...
)
```

而不是标准的 `.testTarget`。这导致需要自定义 runner，且无法利用 XCTest 的测试发现和并行执行。

### 5.3 测试辅助类缺少协议隔离（中等）

`PipelineTestClient`、`FakeAICompletionClient`、`SequencedAICompletionClient` 等测试辅助类定义在测试文件内部，与产品代码耦合。如需在其他 target 中复用，必须复制。

### 5.4 异步测试依赖轮询超时（低）

`waitForMainActor(timeout:condition:)` 使用 20ms 间隔轮询，默认 4 秒超时。在慢速 CI 环境中可能误判。建议使用 Swift Concurrency 的 `withCheckedContinuation` 或 XCTest 的 `XCTestExpectation`。

## 六、UI 架构

### 6.1 零本地化支持（严重）

整个应用 100% 硬编码中文，没有任何 `Localizable.strings` 或 `String(localized:)` 调用。要支持其他语言需要重写所有代码。

### 6.2 零无障碍支持（严重）

没有任何 `.accessibilityLabel()`、`.accessibilityValue()`、`.accessibilityHint()` 或 `@ScaledMetric` 的使用。`ClozeBlankField` 的 `TextField("", ...)` 空标签对 VoiceOver 不可见。

### 6.3 零 Preview 支持（中等）

58 个 SwiftUI View 文件中没有一个 `#Preview` 或 `PreviewProvider`。开发时每次 UI 调整都必须完整启动 App。

### 6.4 字符串分类依赖硬编码比较（中等）

**文件**：`Models/TranslationJob.swift:129-138`、`Models/PracticeLibrarySummary.swift:12-22`

`isLocalFileSource` 和 `isLocalFileLibrary` 通过 `source == "本地文件"` 这样的字符串匹配来分类。如果源文本变更或翻译，逻辑会静默失效。**建议**：使用 enum 或 Bool 属性。

### 6.5 文件扩展名列表重复 3 处（中等）

`.txt`、`.md` 等扩展名列表出现在 `PracticeStore.swift`、`PracticeLibrarySummary.swift`、`ImportView.swift` 各一套。应抽取为单个常量。

### 6.6 强制解包（中等）

StudyStore 中 7 处 `Calendar.date(byAdding: ...)!` 和 ClozeGenerator 中的 `try! NSRegularExpression`。应使用 guard-let 或 do-catch 替代。

### 7.2 项目命名 `whatever`（低）

`Package.swift` 中 product 名称、app 名称、通知标题、目录路径均为 `whatever`，是未替换的占位名。

### 7.3 `@testable import` 用于生产代码（低）

**文件**：`Sources/EnglishClozeCoach/App/EnglishClozeCoachApp.swift:3`

```swift
@testable import EnglishClozeCoach
```

`@testable` 通常只在测试 target 中使用。生产入口不应该依赖测试基础设施。

### 7.4 硬编码的 `-enable-testing` 在所有 target（低）

**文件**：`Package.swift`

```swift

```

应仅在 debug 构建中使用，或通过条件编译控制。

## 建议重构路线

### 第一阶段（紧急）：拆解巨型文件
1. 拆分 `TranslationJobStore` 为 Pipeline + Buffer + Persistence + Store
2. 从 `ContentView` 抽取 `PracticeCoordinator` ViewModel
3. 拆分 `ImportView` 按导入源类型

### 第二阶段（重要）：提升质量基础
4. 迁移测试到 XCTest + 多文件结构
5. 添加协议抽象（`PracticeLibrary`、`QuestionImporter` 等）
6. 消除 `@unchecked Sendable`，重构为显式 Sendable 一致性
7. 统一错误处理模式，消除 `try?` 吞错误

### 第三阶段（改进）：完善工程化
8. 添加本地化框架
9. 添加无障碍标注
10. 添加 SwiftUI Preview
11. 统一硬编码常量，消除魔法数字
12. 重命名 `whatever` → 正式产品名
